Template and Application Pages

Template Pages

In the world of knowledge graphs it is important to distinguish between instance data and model data. Instances such as, for example, a representation of a particular person can be typed according to an ontology (model) as, for example, being of type Agent.

In principle, it is possible to create a page for every instance of type Agent within the knowledge graph manually. However, templates can be used to define generic views which are being automatically applied to entire sets of instance, for example, to all instances of type Agent.

The templateIncludeQuery in the UI Configuration can be used to refine the logic according to which candidate templates are being computed when browsing instances in the knowledge graph. By default only explicit types (i.e. rdf:type) will be selected as templates for the current instance. If none of the candidate templates exist, the system-wide default Template:rdfs:Resource will be used.

To create a new or modify an existing templates, just follow the link within the top of the template editor when editing an instance page (i.e. when switching to the edit mode on an instance page, applicable templates are being listed in the top of the editor).

Application Pages

Application pages are those pages which make up your application, but are not directly associated with any entity in the knowledge graph. Let's say, for example, an dashboard page may provide an aggregated view and actions over some data in the domain, but will not be rendered according to some automatically inherited templates but rather using static markup (which still might be parameterized dynamically).

Difference: Template vs. Application Pages

The fundamental difference between template and pure application pages is their context. In particular, when using the Back-end Templating Syntax and Helper Functions   it is fundamental to understand the meaning of the special ?? variable within SPARQL queries and [[{{documentation}}]][[this]][[{{/documentation}}]] within the template pages, which will be set to the the IRI of the current RDF resource.

On application pages, however, the native context is useless (since application pages do not have corresponding RDF entity in the knowledge graph) and - if at all - dynamic context parameters need to be passed externally, for example, using parameterized includes or url parameters.
