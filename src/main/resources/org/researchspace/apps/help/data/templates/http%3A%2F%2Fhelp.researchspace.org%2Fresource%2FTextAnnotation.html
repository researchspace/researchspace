Text Annotation Components

Text annotation component loads the document from the storage and provides UI for viewing/changing/removing annotations of different types.

Text annotation workspace uses <semantic-context> declaration to determine repository to read metadata about the document, fetch annotation data and post new annotations. Other data usage like auto-suggestion for annotation editor inherits this semantic context but may be overridden in user-provided markup.

The documents are identified by IRI and expected to follow the structure for files uplodaded through file manager. An uploaded document consists of:

  • LDP resource with associated metadata and a file name.
  • HTML-file with the same file name in the storage (specified in the workspace attributes). The file may have an arbitrary nesting structure with the following limitations:
    • Block tags: div/p/section/blockquote/pre h* table/tbody/thead/tr/th/td
    • Inline tags: span
    • Mark tags: b/i/u/strong/em a code
    • Block tags may contain either only other block tags or only inline/mark tags but not both at the same time.

Custom Sidebar Tabs

The workspace sidebar can be extended with custom tabs alongside the default Annotations tab. This is useful for embedding additional context such as IIIF viewers, metadata panels, or any other template-driven content next to the annotated document.

Configuration

Two optional attributes on <rs-text-annotation-workspace> control custom tabs:

Attribute Type Description
sidebar-tabs JSON string A JSON array of tab definitions. Each element is an object with the following properties:
  • key — unique identifier (restricted to [a-zA-Z0-9_-])
  • label — display text for the tab (non-empty string)
  • iconUrl — (optional) URL for the tab icon; only http://, https://, and relative paths are permitted
default-sidebar-tab string The key of the custom tab to pre-select on mount. When absent or unmatched, the Annotations tab is selected by default.

Tab Content Templates

Each custom tab's content is provided as a child <template> element inside the workspace. The template id must follow the pattern sidebar-{key}, where {key} matches the tab definition's key.

Inside the template, the variable {{iri}} is bound to the document IRI, allowing queries and components to reference the current document.

Input Validation

The component validates sidebar-tabs input to prevent malformed or unsafe configuration:

  • JSON parse errors produce a descriptive error message.
  • The parsed value must be an array; each element must be an object.
  • key is restricted to [a-zA-Z0-9_-] to prevent template-ID injection.
  • label must be a non-empty string.
  • iconUrl is validated via isSafeUrl() — only http://, https://, and relative paths are permitted; schemes such as javascript:, data:, and vbscript: are rejected.

Backward Compatibility

When sidebar-tabs is not provided, the component behaves identically to before — no custom tabs appear and the annotation list renders as the only sidebar content.

Event Bus Integration

The workspace can communicate bidirectionally with custom sidebar tab templates (or any other component on the page) via the platform event bus. This requires setting the id attribute on the workspace.

Events

Event Type Direction Data Description
TextAnnotation.FocusAnnotation Incoming (template → workspace) { annotationIri: string } Focus and highlight the annotation with the given IRI. Scrolls the editor to bring it into view.
TextAnnotation.AnnotationFocused Outgoing (workspace → template) { annotationIri: string } or {} Fired when an annotation is focused (from editor click, sidebar click, or incoming event). Empty data when focus is cleared.

When id is not set, no events are listened to or emitted — the workspace behaves identically to before.
