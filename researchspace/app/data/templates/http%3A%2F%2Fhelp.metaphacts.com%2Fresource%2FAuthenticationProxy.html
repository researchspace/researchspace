Authentication Proxy

Sometimes there appears a need to access from the metaphacts platform an external web source (e.g., a third-party REST API, images, files etc.), which imposes additional access restrictions like protection through HTTP basic authentication or strict Cross-Origin Resource Sharing (CORS) configurations. For this purpose, the platform provides a mechanism to proxy such requests including basic authentication. It is possible to configure a proxy servlet, which will forward all HTTP requests it receives to a specific remote target URL. Such a proxy servlet for accessing a remote URL targetUri requiring separate authentication transforms a request of the form [platform URL]/proxy/[proxy ID]/path/to/resource into [targetUri]/path/to/resource and adds the authentication header.

Proxy configuration parameters

The targetUri and the login credentials are passed via properties in two ways:
  • Using the file config/proxy.prop.
  • Using Java system properties.
The following properties are supported:
  • config.proxy.[proxy ID].targetUri: the target URL to redirect the requests to.
  • config.proxy.[proxy ID].preserveCookies: preserve HTTP COOKIES as is. Default: false.
  • config.proxy.[proxy ID].forwardip: enable forwarding of the client IP using X-Forwarded-For and X-Forwarded-Proto HTTP headers. Default: true.
  • config.proxy.[proxy ID].preserveHost: preserve HTTP HOST parameter as-is. Default: false.
  • config.proxy.[proxy ID].handleRedirects: follow HTTP redirects. Default: false.
  • config.proxy.[proxy ID].socketTimeout: the socket read timeout (millis). Default: -1.
and two alternative options to pass the login credentials
  • config.proxy.[proxy ID].loginName and
  • config.proxy.[proxy ID].loginPassword: login and password for basic authentication on the target web site.
OR
  • config.proxy.[proxy ID].loginBase64: base64-encoded "login:password" pair. This option is only applied when login and password are not provided explicitly. Note: this option exists only to avoid defining login and password in the configuration file as plain text, but does not provide secure data interchange.
Instead of specifying the proxyTargetUri and credentials in the properties file, it is also possible to specify these variables using Java system properties (-Dconfig.proxy.[proxy ID].targetUri=https://myservice/), for example, during deployment time.

Access control

In the platform it is possible to configure multiple proxy servlets for different remote target URIs as well as for different access modes of the the same API: for example, the path /proxy/[proxy ID]/path/to/resource can be used to access the targetUri with the standard user login and password, while /proxy/[proxy ID-admin]/path/to/resource will access the same targetUri with the administrator credentials. In the metaphacts platform, access to a specific instance of the authentication proxy servlet with the ID [proxy ID] can be regulated using the standard Apache Shiro security mechanism (see basic system configuration). The ACL permission for each instance of the proxy servlet with the ID [proxy ID] is defined by the string proxy:[proxy ID].
